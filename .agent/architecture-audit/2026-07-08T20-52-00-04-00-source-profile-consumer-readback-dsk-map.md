**Timestamp:** `2026-07-08T20-52-00-04-00`

## Goal

Document the current DSK/domain surface for `PhantomCommand` and narrow the next safe implementation to source-profile ownership, DOM-free parity fixtures, additive GameHost sourceProfile readback, and build-gate integration.

## Active route

```txt
index.html
  -> game.html
  -> Three.js CDN
  -> inline smooth-ring-handoff-v6 construct runtime
```

## Current interaction loop

```txt
open index.html
  -> click Start or Open Scene
  -> game.html creates renderer, scene, camera, lights, materials, HUD, input, and construct state inline
  -> smooth-ring-handoff-v6 constants generate rings and pieces inline
  -> construct(seq) animates 92 pieces through ringStartTimes and PART_STAGGER
  -> user pans, zooms, skips, or restarts
  -> HUD and window.GameHost expose construct-only state
```

## Main architectural finding

The live `game.html` route is stable but source-opaque. It exposes useful `GameHost` state, but that state is generated by the same inline runtime that renders the scene, so there is no independent proof that the source profile, descriptor math, timeline, and GameHost readback all agree.

## Next safe ledge

```txt
PhantomCommand SourceProfile Consumer Readback + Fixture Build Gate
```

Build source profile modules and fixtures first. Do not extract renderer code, add RTS gameplay, or create scenario bootstrap until sourceProfile parity and GameHost consumer readback are proven.
